The Duolingo pre-commit hook automatically excludes certain files and directories to prevent formatting errors and improve performance. You can also add custom exclusions for your project.

## Global Exclusions

These files and directories are **always excluded** by the hook and cannot be formatted:

### Excluded Directories

From `entry.ts:485-486`:

```typescript
const FOLDER_EXCLUDES = [".claude/skills/.generated", "build", "node_modules"];
```

### Excluded Files

From `entry.ts:487-491`:

```typescript
const FILE_EXCLUDES = [
  "AGENTS.md", // Generated by sync-ai-rules pre-commit hook
  "copilot-instructions.md", // Generated by sync-ai-rules pre-commit hook
  "gradlew",
];
```

| File | Reason |
|------|--------|
| `AGENTS.md` | Generated by sync-ai-rules hook |
| `copilot-instructions.md` | Generated by sync-ai-rules hook |
| `gradlew` | Gradle wrapper script (should not be modified) |

### Exclusion Pattern

The global exclusion regex from `entry.ts:492-494`:

```typescript
return RegExp(
  `(^|/)((${FOLDER_EXCLUDES.join("|")})/)|(${FILE_EXCLUDES.map(f => f.replace(/\./g, "\\.")).join("|")})$)`,
);
```

This matches:
- Any path containing an excluded directory (e.g., `src/build/file.js`, `build/output.js`)
- Any file with an excluded filename (e.g., `./AGENTS.md`, `scripts/gradlew`)

## Tool-Specific Exclusions

Some formatters have additional exclusions:

### Minified JavaScript Files

From `entry.ts:14`:

```typescript
const MINIFIED_JS_REGEX = /\bmin\b|\.(custom|pack)\./;
```

Excludes files matching:
- `*.min.js` - Minified JavaScript
- `*.custom.js` - Custom builds
- `*.pack.js` - Packed JavaScript

Affects:
- Prettier (JS files only)
- ESLint

### Protobuf Files (Shell Formatter)

From `entry.ts:426`:

```typescript
[HookName.Shfmt]: {
  // pre-commit's `types: [text]` config option sometimes has false positives,
  // and removing a binary .proto file's trailing newline may corrupt it
  exclude: /\.proto$/,
```

Excludes `.proto` files from shell formatting to prevent binary file corruption.

## Custom Exclusions

You can add project-specific exclusions using pre-commit's `exclude` key:

### Exclude Specific Files

```yaml .pre-commit-config.yaml
- repo: https://github.com/duolingo/pre-commit-hooks.git
  rev: 1.13.3
  hooks:
    - id: duolingo
      exclude: ^\.github/workflows/
```

Excludes all files in `.github/workflows/` directory.

### Exclude Multiple Patterns

```yaml .pre-commit-config.yaml
- repo: https://github.com/duolingo/pre-commit-hooks.git
  rev: 1.13.3
  hooks:
    - id: duolingo
      exclude: |
        (?x)^(
          migrations/.*\.py|
          legacy/.*|
          vendor/.*|
          .*\.generated\.(js|ts)
        )$
```

Excludes:
- All Python files in `migrations/`
- Everything in `legacy/` directory
- Everything in `vendor/` directory
- Any file ending with `.generated.js` or `.generated.ts`

### Exclude by File Type

Use pre-commit's `types` or `types_or` keys:

```yaml .pre-commit-config.yaml
- repo: https://github.com/duolingo/pre-commit-hooks.git
  rev: 1.13.3
  hooks:
    - id: duolingo
      exclude_types: [markdown, json]
```

Excludes Markdown and JSON files.

<Note>
  The `exclude_types` approach is cleaner than regex patterns for common file types.
</Note>

## File Filtering Logic

From `entry.ts:570-576`:

```typescript
const includedSources = sources.filter(
  source =>
    include.test(source) &&
    !GLOBAL_EXCLUDES.test(source) &&
    !(exclude && exclude.test(source)),
);
```

A file is processed if:
1. ✅ It matches the tool's `include` pattern (e.g., `/\.py$/` for Python tools)
2. ✅ It does NOT match `GLOBAL_EXCLUDES`
3. ✅ It does NOT match the tool's `exclude` pattern (if any)

## Common Exclusion Patterns

<AccordionGroup>
  <Accordion title="Generated Code">
    ```yaml
    exclude: |
      (?x)^(
        .*\.generated\..*|
        .*_pb2\.py|
        src/gen/.*
      )$
    ```
    
    Excludes:
    - Files with `.generated.` in the name
    - Python protobuf files (`*_pb2.py`)
    - Everything in `src/gen/` directory
  </Accordion>
  
  <Accordion title="Third-Party Code">
    ```yaml
    exclude: ^(vendor|third_party|external)/
    ```
    
    Excludes directories containing vendored dependencies.
  </Accordion>
  
  <Accordion title="Legacy Code">
    ```yaml
    exclude: ^(legacy|deprecated|old)/
    ```
    
    Excludes legacy code that you don't want to reformat.
  </Accordion>
  
  <Accordion title="Test Fixtures">
    ```yaml
    exclude: |
      (?x)^(
        tests/fixtures/.*|
        test/data/.*|
        testdata/.*
      )$
    ```
    
    Excludes test fixture files that should remain unformatted.
  </Accordion>
</AccordionGroup>

## Best Practices

<CardGroup cols={2}>
  <Card title="Minimize Exclusions" icon="filter">
    Only exclude files that:
    - Are generated automatically
    - Come from third-party sources
    - Would break if formatted
    
    Avoid excluding code you maintain.
  </Card>

## Debugging Exclusions

To see which files would be processed:

```bash
# Dry run with verbose output
pre-commit run duolingo --all-files --verbose

# Show files that match pre-commit's file filter
git ls-files | pre-commit run duolingo --files
```

<Tip>
  Use `--verbose` to see why files are being skipped.
</Tip>

## Excluding Individual Lines

Some formatters support inline disable comments:

<Tabs>
  <Tab title="ESLint">
    ```javascript
    // eslint-disable-next-line
    const x = eval('dangerous code');
    ```
  </Tab>
  
  <Tab title="Prettier">
    ```javascript
    // prettier-ignore
    const matrix = [
      1, 0, 0,
      0, 1, 0,
      0, 0, 1
    ];
    ```
  </Tab>
  
  <Tab title="Ruff">
    ```python
    import os  # noqa: F401
    ```
  </Tab>
  
  <Tab title="Black">
    ```python
    # fmt: off
    matrix = [
        1, 0, 0,
        0, 1, 0,
        0, 0, 1,
    ]
    # fmt: on
    ```
  </Tab>
</Tabs>

<Warning>
  Use inline disable comments sparingly. They often indicate code that should be refactored or placed in a dedicated file that can be excluded entirely.
</Warning>
